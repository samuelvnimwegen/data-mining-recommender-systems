\section{Introduction}
This report presents a movie recommender system built from \texttt{movies.csv} and \texttt{ratings\_train.csv}. The project has two goals: first, to build and compare collaborative filtering and matrix factorization models; second, to generate ten recommendations for every user in \texttt{ratings\_test.csv}. The full implementation is available in the linked Git repository.

\section{Data Preprocessing and Feature Engineering}
Before training any model, the pipeline validates the raw files and removes rows that can break training or bias results. Missing identifiers, invalid numeric values, duplicates, and orphan ratings are removed so every rating references a known movie. This keeps matrix construction stable and avoids silent errors.

Movie metadata is transformed into model-ready features. Genre strings are split and one-hot encoded into \texttt{genre\_*} columns, and the release year is extracted from titles when present. Optional TF-IDF genre features are also supported for richer semantic signals. On the ratings side, user-centered normalization and timestamp-derived features are available, including optional time-decay weighting to emphasize recent interactions.

\section{Task 1: Recommender Models and Evaluation}
The implementation includes ItemKNN (memory-based collaborative filtering), SVD (matrix factorization), and LightFM (hybrid matrix factorization with item metadata). This satisfies the requirement to compare at least one collaborative filtering model and one matrix factorization model, while also including a hybrid model for sparse settings.

Offline evaluation is performed on a deterministic user-aware train/validation split. Accuracy is measured with RMSE and MAE. Ranking quality is measured with Precision@K, Recall@K, and NDCG@K using a relevance threshold. Beyond-accuracy analysis includes popularity-based Novelty@K, Diversity@K, ItemCoverage@K, IntraListSimilarity@K, ItemToHistoryDistance@K, and Serendipity@K.

The trade-off pattern is consistent with recommendation literature. ItemKNN is easy to train but can be slower at inference because it depends on neighborhood lookups. SVD and LightFM require heavier offline training but are efficient at serving predictions once factors are learned. LightFM usually offers a stronger personalization-versus-coverage balance in sparse data because metadata links users to items even when interaction overlap is limited.

\subsection{Feature Influence in LightFM}
To inspect which engineered features help or hurt model quality, the codebase includes an ablation-based feature influence analysis. The method first trains a baseline LightFM model with all features, then repeats training while setting one feature column to zero at a time. The selected metric difference (for example RMSE delta) is recorded as an influence score. Positive values indicate that removing the feature worsens performance, so the feature has a positive contribution. Negative values indicate the opposite.

\section{Task 2: Predictions for Specific Users}

The notebook \texttt{notebooks/08\_predictions\_usage.ipynb} runs this export end-to-end. Cold-start users are handled through a fallback branch that ranks movies with a Bayesian popularity score and optional genre preference blending. Known users with enough interactions are served by the personalized model path. This routing prevents failures for unseen users while preserving personalized quality where history exists.
